# Batch 9 — P5 Cascade Engine

**Scope**: `lip/p5_cascade_engine/` (9 Py files, ~1.1k LOC).
**HEAD**: `2b32314`
**Reviewer focus**: Corridor cascade ordering, fee recomposition traps, canonical constant compliance (300 bps fee floor), intervention optimizer correctness, entity resolution BIC→corporate mapping, graph algorithm correctness (Brandes betweenness, BFS propagation).

## Summary

The P5 cascade engine is a well-designed supply chain risk module: BIC→corporate entity resolution, BFS cascade propagation with probability multiplication, greedy intervention optimizer (weighted set cover), Brandes betweenness centrality, and C3/C5 integration bridges. The code is clean, the algorithms are correct, and the data structures are appropriate. The **critical issue** is that `INTERVENTION_FEE_RATE_BPS = 200` directly violates the CLAUDE.md canonical fee floor of 300 bps — this constant is used for cascade intervention cost estimation, and a 200 bps estimate means the optimizer underestimates the true cost of bridge loans by 33%, making interventions appear more cost-efficient than they actually are. The cascade propagation's bottom-up CVaR pass has an O(n²) scan pattern (each node scans all nodes to find children) which is fine for small cascade graphs but will be a problem at scale.

## Notable positives (preserve)

- **Brandes betweenness centrality (corporate_graph.py:97-144) is textbook-correct.** Proper Brandes (2001) implementation with BFS predecessor tracking, sigma counting, and normalized [0,1] output. O(VE) complexity is optimal for unweighted graphs.
- **Entity resolver correctly handles intra-corporate filtering.** `src_corp == tgt_corp` payments are excluded (line 45-46), preventing self-loops in the corporate graph that would create false cascade amplification.
- **Corporate feature vector is well-documented.** 8-dimensional feature vector with each dimension named, described, and indexed in the module docstring. HHI supplier concentration and volume-weighted failure rate are the right signals for cascade risk.
- **CascadeSettlementTrigger has correct multi-filter pipeline.** dependency_score → BIC resolution → intra-corporate → CVaR threshold — four filters in the right order, each with clear semantics.
- **Intervention optimizer uses correct greedy weighted set cover.** The (1 - 1/e) approximation guarantee is documented. The `_get_descendants` BFS correctly marks protected subtrees to prevent double-counting of prevented value.
- **StressCascadeBridge keeps CascadeGraph immutable.** The corridor→corporates mapping is injected at construction, not derived from the graph at runtime. This is the right separation of concerns.
- **Constants are documented with authority tags.** Every constant has a docstring and the module header says "All thresholds require QUANT sign-off" — good governance intent (even if B9-01 violates the fee floor).
