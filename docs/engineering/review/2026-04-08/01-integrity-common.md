# Batch 1 — Integrity & Common

**Scope**: `lip/common/` (23 files, ~4.6k LOC) and `lip/integrity/` (8 files, ~2.0k LOC).
**HEAD**: `2b32314`
**Reviewer focus**: crypto, UETR dedup, shared utilities, integrity-shield primitives, canonical constants.

## Summary

Two **Critical** findings, both in load-bearing security primitives that other components import. The crypto module advertises a FIPS power-on self-test that does not exist in code, and the UETR retry tracker — the GAP-04 double-funding control — runs in production with a 30-minute window instead of the canonical 45-day buffer, and silently ignores the Redis client argument so multi-replica deployments cannot share state. The integrity shield modules are well-designed in shape but contain several "warn-only" controls that look like gates but never actually block, plus a canonical-JSON serialiser that loses Decimal/float distinctions.

## Findings

| ID | Sev | File:line | Category | Finding | Suggested fix |
|---|---|---|---|---|---|
| B1-01 | **Critical** | `lip/common/uetr_tracker.py:28`, `lip/pipeline.py:159` | Domain invariant / control bypass | `UETRTracker.__init__` defaults `ttl_seconds=1800` (30 min). Production pipeline (`pipeline.py:159`) calls `UETRTracker()` with no override, so the GAP-04 retry-detection window in prod is **30 minutes**. CLAUDE.md and `constants.py:96` set `UETR_TTL_BUFFER_DAYS = 45`. A bank operationally re-keying a payment 31 minutes later → second loan funded. *Conflicts with CLAUDE.md: "UETR TTL buffer: 45 days".* | Default `ttl_seconds = UETR_TTL_BUFFER_DAYS * 86400`; remove the magic 1800; assert at construction that ttl ≥ minimum maturity. |
| B1-02 | **Critical** | `lip/common/uetr_tracker.py:28-31` | Silent feature stub | `redis_client` parameter is accepted, stored on `self._redis`, and **never read anywhere in the class**. The dedupe is in-memory only. Multi-replica C7/C5 deployments will not share state — each pod is its own dedupe domain → **double funding under any horizontal scale**. The docstring "in production this should be backed by a distributed store" is aspirational; the parameter is dead code that creates a false sense of safety. | Either implement Redis-backed `_processed`/`_tuple_index` (write-through, lookup-first) or **remove the parameter** and raise loudly at startup if more than one replica is detected. Half-implemented is worse than missing. |
| B1-03 | **Critical** | `lip/common/encryption.py:23-25` | Fabricated control / regulatory misrepresentation | Comment claims `_HMAC_SELF_TEST_DIGEST` is "Verified at module import; mismatch raises RuntimeError" per FIPS 140-2 §4.9.1. Grep across `lip/` shows the constant is **defined but never used** — there is no power-on self-test code anywhere. A regulatory artifact (data card, REX submission) that cites FIPS conformance would be fabricated. *Conflicts with CLAUDE.md: REX rule "will refuse to mark a model deployment-ready without …"; same standard applies to crypto controls.* | Either implement the self-test (compute HMAC of a vector at import, compare against the constant, raise on mismatch, log success) or **delete the constant and the comment**. Do not ship a fake FIPS claim. |
| B1-04 | High | `lip/common/uetr_tracker.py:90-92, 132-134` | Fail-open security control | Both the index update and the tuple-match path swallow `Exception` and continue. A malformed `context` (e.g. `amount=None`, non-Decimal-parseable string) causes `_is_tuple_match` to return `False` → duplicate is allowed. The whole point of GAP-04 is to fail **closed**. | Validate `context` upfront (pydantic model) and raise on malformed input; or, in the catch, return `True` (treat-as-retry) so the worst case is an over-blocked legitimate payment, not a double-fund. |
| B1-05 | High | `lip/common/encryption.py:21` | Crypto parameter below current best practice | `_PBKDF2_ITERATIONS = 100_000`. OWASP 2023 password-storage cheat sheet recommends ≥ 600,000 for PBKDF2-HMAC-SHA256. Used by `derive_key_from_passphrase` to produce AES-256 KEKs. Not catastrophic, but for a system that signs licensee tokens and encrypts at rest, the parameter is a decade old. | Bump to ≥ 600,000, or migrate to `argon2-cffi` (memory-hard). Add the iteration count to the derived key blob so future bumps are decryption-compatible. |
| B1-06 | High | `lip/integrity/pipeline_gate.py:7-8, 92-104` | Control that doesn't gate | `IntegrityGate.check()` is "warn-only" by design. `gate_passed=False` only logs a warning; nothing in the pipeline blocks on it. The class is named `IntegrityGate` but is a metric, not a gate. CLAUDE.md REX rule says nothing ships without evidence — yet GPL contamination on a dependency would only emit `logger.warning(...)`. | Wire `gate_passed=False` into `LIPPipeline.__init__` as a hard raise (or, if that's too disruptive, a startup-mode flag with `enforce=True` default in prod). The warn-only mode should be opt-in for local dev only. |
| B1-07 | High | `lip/integrity/pipeline_gate.py:44, 49` | Dead config / silent feature stub | `model_evidence_max_age_hours: int = 720` is stored on `self._max_age` but **never used in `check()`**. Looks like an evidence-freshness check that was stubbed and forgotten. Same anti-pattern as B1-02. | Either implement the evidence-age check (call `claims_registry.list_stale(self._max_age)`) or delete the parameter. |
| B1-08 | Medium | `lip/common/encryption.py:54-79` | Crypto construction (not best practice) | `hash_identifier` uses raw `SHA-256(value || salt)`. Length-extension is not a practical threat for this comparison-only use case, but **HMAC-SHA256(salt, value)** is the standard primitive for keyed pseudonymization and is what an external auditor will look for. | Switch to `hmac.new(salt, value.encode("utf-8"), sha256).hexdigest()`. Same output shape, cleaner provenance, easier to defend in a CIPHER review. |
| B1-09 | Medium | `lip/integrity/evidence.py:122-131`, `lip/integrity/compliance_enforcer.py:78-86` | Canonical-JSON loses type info | `_canonical_json` and `compute_proof_of_freshness` use `json.dumps(..., default=str)`. A `Decimal("1.50")` and the float `1.5` and the string `"1.5"` all serialise to `"1.5"` (or close to it) — type changes between live data and stored evidence will not invalidate the hash. For an integrity primitive, that is too loose. | Use a stricter canonicalisation (RFC 8785 JCS), or pre-serialise Decimals as `{"__decimal__": "1.50"}` and reject non-JSON-native types at the boundary. Add a regression test that round-trips `Decimal` ≠ `float` produces different hashes. |
| B1-10 | Medium | `lip/integrity/compliance_enforcer.py:53-75`, used by `vendor_validator.py` | Detector calibration | Boilerplate detector uses **character 3-grams** with a 90% threshold. For natural-language compliance reports this is too granular — standard phrases ("This report covers the period…", "No material findings…") drive baseline overlap toward 0.4-0.6, and legitimate reports that share a template can cross 0.9. False positives will push operators to dial down the threshold, defeating the control. | Use word-level 5-shingles, or TF-IDF cosine similarity weighted by document frequency. Add a calibration test against a corpus of known-distinct reports to set the threshold empirically. |
| B1-11 | Medium | `lip/integrity/compliance_enforcer.py:111-141` | Subtle binding gap | `wrap_report_generation(generator, live_data, *args, **kwargs)` calls `generator(*args, **kwargs)` and binds the resulting evidence to `live_data` — but **`live_data` is never passed to the generator**. The "proof of freshness" only proves the caller hashed *some* dict; it does not prove the report was actually computed from that dict. A lazy or buggy caller can pass `live_data=current_state` and a generator that reads from anywhere. | Change signature to `generator(live_data, *args, **kwargs)` so `live_data` is the only data input the generator receives. Or, document the gap loudly in the docstring (currently silent). |
| B1-12 | Medium | `lip/common/constants.py:163` | Risk-up default | `MIN_LOAN_AMOUNT_USD = Decimal("500000")` is documented as the "legacy default for unlabelled / unknown class". Defaulting to the **lowest** tier on unknown class is the wrong direction — unknown class should attract the **highest** floor (the Class A tier, $1.5M), or refuse the offer. *QUANT call.* | Change default to `MIN_LOAN_AMOUNT_CLASS_A_USD` (or raise a `ClassUnknownError` and let C7 decline). Add a unit test pinning the default. |
| B1-13 | Medium | `lip/common/constants.py:21-22, 92-93` | Constant duplication | `LATENCY_P99_TARGET_MS = 94` and `LATENCY_SLO_MS = 94` are two names for the same value. Two names invite future drift (one will be bumped, the other forgotten). | Keep `LATENCY_SLO_MS` (the canonical name in CLAUDE.md), make `LATENCY_P99_TARGET_MS = LATENCY_SLO_MS`, then deprecate. Same applies to `PHASE_1_BPI_FEE_SHARE` vs `PLATFORM_ROYALTY_RATE`. |
| B1-14 | Medium | `lip/common/constants.py:6` | Marketing constant in security file | `MARKET_SIZE_USD = 31_700_000_000_000` is a slide-deck number (2024 FXC Intelligence). It does not belong next to `FEE_FLOOR_BPS` and `LATENCY_SLO_MS`. Mixing business comms with money-control constants raises the chance someone "tweaks a constant" and hits the wrong one. | Move to `lip/common/market_metadata.py` or a docs file. `constants.py` should contain only values that affect runtime arithmetic. |

## Notable positives (preserve)

- `encryption.py` uses `hmac.compare_digest` for signature verify (B1-03 aside, the verification path itself is correct).
- `evidence.py` `sign_claim` refuses to sign a claim with empty `evidence_ids` — exactly the right shape for a "no claim without evidence" rule.
- `breach_protocol.py` `PROHIBITED_ASSURANCES` list and the structured DETECTED → SENT state machine are a thoughtful response to a specific failure mode.
- `constants.py` is unusually well-commented: every QUANT-controlled value has a date, sign-off owner, and rationale. This is rare and should not be eroded.
- `state_machines.py` imports class-day constants from `constants.py` instead of redefining them — no drift.
- `partial_settlement.py` defaults to `REQUIRE_FULL` (the conservative policy) and requires explicit opt-in for `ACCEPT_PARTIAL`.
