# Batch 5 — C3 Repayment Engine + Rust State Machine

**Scope**: `lip/c3_repayment_engine/` (8 files, ~1.8k LOC), `lip/c3/` (2 Py files, ~690 LOC), `lip/c3/rust_state_machine/` (5 Rust files, ~1.0k LOC).
**HEAD**: `2b32314`
**Reviewer focus**: Money arithmetic (Decimal vs float), maturity constants, state transition correctness, settlement rail handling, dual-language taxonomy drift, idempotency.

## Summary

The Rust state machine is **the best-engineered component reviewed so far** — exhaustive transition table, no panics, no `unwrap` outside tests, clean PyO3 bindings. The Python repayment engine surrounding it is functional but has a recurring theme: **every safety control has a silent in-memory fallback that defeats the control's purpose under infrastructure failure or horizontal scale**. The Redis SETNX idempotency guard falls back to a per-pod `set()` that allows double-repayment across replicas. The UETR mapping falls back to in-memory with no TTL enforcement. The corridor buffer is unsynchronized and uses wall-clock time. Most critically, the rejection taxonomy is maintained in **both Python and Rust** with no automated drift detection — a code-level BLOCK adding a new compliance-hold code to one language but not the other is a straight path to bridging a compliance-held payment (EPG-19 violation). The settlement handlers duplicate ISO 20022 field extraction that already exists in the Rust crate, and every handler wraps its core logic in a bare `except Exception` that swallows parse errors and returns a fallback — fail-open on the settlement confirmation path.

| B5-01 | **Critical** | `lip/c3_repayment_engine/rejection_taxonomy.py` + `lip/c3/rust_state_machine/src/taxonomy.rs` | Dual-maintained taxonomy — drift = EPG-19 violation | The rejection code→class mapping is maintained **independently** in Python (dict literal, ~50 codes) and Rust (match arms, ~50 codes). No automated check verifies they produce the same classification for the same input. If a new compliance-hold code (e.g. a future SWIFT code for court-ordered freeze) is added to Python's `BLOCK` class but missed in Rust's `Block` match arm, the Rust path classifies it as unknown (returns `Err`) and the Python bridge's fallback path (B5-02) silently uses the Python classification — but any direct Rust caller (future C5 streaming consumer, batch job) would **not block the payment**. Given EPG-19's "never bridge a compliance hold" invariant, this is a privilege-escalation path via taxonomy drift. *Conflicts with CLAUDE.md: EPG-19 defense-in-depth requires both layers to agree.* | Single source of truth: either (a) generate the Rust match arms from the Python dict at build time (build.rs reads a shared JSON), or (b) generate the Python dict from the Rust crate via PyO3 (already exposed as `classify_rejection_code`). Add a CI test that iterates every code in both sources and asserts identical classification. |
| B5-02 | **Critical** | `lip/c3/state_machine_bridge.py:38-52` | Silent fallback to pure-Python state machine | `try: from lip_c3_rust_state_machine import …` falls back to a pure-Python reimplementation with `warnings.warn(...)`. In production, a missing `.so` (broken wheel build, Alpine vs manylinux mismatch, Docker layer cache miss) silently downgrades the entire C3 repayment path to the Python fallback — which is **not automatically tested against the Rust implementation for behavioral equivalence**. The fallback state machine may have different edge-case behavior (e.g. the Rust version returns `Err` for unknown rejection codes; the Python fallback returns `False` from `is_dispute_block`). For a payment state machine, silent behavioral divergence is a correctness violation. | Either: (a) make the Rust import **mandatory** in production (`LIP_REQUIRE_RUST_SM=true`, raise on import failure), with the Python fallback gated behind an explicit `LIP_ALLOW_PYTHON_FALLBACK=true` env var; or (b) add a comprehensive equivalence test suite that runs every transition and every rejection code through both implementations and asserts identical results. Currently uses `warnings.warn` which is invisible in production (warnings are often filtered). |
| B5-03 | High | `lip/c3_repayment_engine/repayment_loop.py:~claim_repayment` | Redis SETNX fallback defeats idempotency | `_claim_repayment` uses Redis `SETNX` for distributed idempotency (correct pattern). On Redis failure, it falls back to `self._claimed_local.add(uetr)` — a per-pod `set()`. With N replicas, each pod has its own `_claimed_local`, so the same UETR can be claimed on N pods simultaneously → **double repayment**. The fallback is worse than failing: it creates a false sense of safety while allowing the exact scenario SETNX was designed to prevent. Same anti-pattern as B1-02, B2-11, B3-04, B4-05. | On Redis failure, **refuse the repayment** (fail closed). Log an alert, let the next retry cycle pick it up when Redis recovers. The in-memory fallback should only exist for single-replica dev/test mode, gated behind an explicit flag. |
| B5-04 | High | `lip/c3_repayment_engine/settlement_handlers.py:*` | Bare `except Exception` on every settlement handler | All 5 rail handlers (SWIFT camt.054, FedNow, RTP, SEPA, Buffer) wrap their core extraction + state-transition logic in `except Exception` blocks that return a fallback/default result. A malformed ISO 20022 message, a `KeyError` from a missing field, or a programmer bug (e.g. `AttributeError`) all get swallowed. On the settlement confirmation path, swallowing a parse error means **treating a failed parse as a non-event** — the payment stays in its current state instead of transitioning, and no alert fires. The Rust crate already provides `extract_camt054_fields` and `extract_pacs008_fields` with proper error propagation. | Narrow each handler's catch to the specific exceptions expected from the ISO 20022 parsing path (`json.JSONDecodeError`, `KeyError`, `ValueError`). Let `AttributeError`, `TypeError`, and other programmer bugs propagate. Better yet, delegate field extraction to the Rust crate's PyO3-exposed functions (already available) and let `PyValueError` propagate on parse failure. |
| B5-05 | High | `lip/c3_repayment_engine/settlement_handlers.py` + `lip/c3/state_machine_bridge.py` | Duplicated ISO 20022 extraction logic | The settlement handlers in Python re-implement camt.054 and pacs.008 field extraction (UETR, amount, currency, rejection code) that **already exists in Rust** and is exposed via PyO3 (`extract_camt054_fields`, `extract_pacs008_fields` in `pyo3_bindings.rs:196-260`). The Python versions use hand-written dict navigation; the Rust versions use `serde_json` with structured fallback paths and proper error types. Two implementations of the same ISO 20022 parsing = two sources of divergence, two maintenance surfaces, two places where a field rename or nested-path change must be applied. | Replace the Python settlement handlers' inline extraction with calls to the Rust-exposed `extract_camt054_fields` / `extract_pacs008_fields`. If the Rust crate is unavailable (B5-02 fallback), the Python fallback should use a **single** shared extraction function, not per-handler inline parsing. |
| B5-06 | High | `lip/c3_repayment_engine/corridor_buffer.py:*` | No thread safety + wall-clock time | `CorridorBuffer` uses `time.time()` (wall clock) for timestamp recording. Wall clock is subject to NTP jumps — a backward jump can make recent observations appear old, causing premature eviction and P95 underestimation. The buffer has no locking despite being accessed from the repayment loop (which runs concurrently with settlement handlers). A concurrent `record()` and `estimate_p95()` on the same corridor can race on the underlying list. Additionally, `numpy` is imported for a single `np.percentile` call that could use `statistics.quantiles` (stdlib, Python 3.8+). | Use `time.monotonic()` for timestamps. Add a `threading.Lock` per corridor (or a single lock if contention is low). Replace `numpy.percentile` with `statistics.quantiles` to eliminate the numpy dependency in this module. |
| B5-07 | Medium | `lip/c3_repayment_engine/uetr_mapping.py:*` | In-memory fallback has no TTL | `UETRMapping` maps RTP `EndToEndId` → UETR. Redis-backed with in-memory fallback. The in-memory dict has **no TTL enforcement** — entries accumulate indefinitely until process restart. For a long-running pod handling high RTP volume, this is a memory leak. The Redis path uses `EXPIRE` correctly; the in-memory path does not mirror this. | Add a TTL-based eviction to the in-memory fallback (e.g. `OrderedDict` with timestamp, periodic cleanup), or refuse to operate without Redis (same recommendation as B5-03). |
| B5-08 | Medium | `lip/c3_repayment_engine/repayment_loop.py:~trigger_repayment` | Fee decomposition uses float multiplication | The phase-aware fee decomposition (`platform_take_rate_pct` applied to `fee_bps`) appears to use float arithmetic for the take-rate multiplication. In a system where `fee_bps` is a `Decimal` (from `constants.py`), multiplying by a float take-rate silently converts to float, losing Decimal precision. For a 300 bps fee with a 30% take rate, the rounding error is sub-cent, but for high-value loans ($50M+) or unusual take rates, float drift can produce a fee split that doesn't sum to the total. *QUANT concern.* | Ensure `platform_take_rate_pct` is stored as `Decimal` (not float) in the license token (see also B3-07). Perform all fee arithmetic in `Decimal` with explicit rounding mode (`ROUND_HALF_EVEN` per banking convention). |
| B5-09 | Medium | `lip/c3/state_machine_bridge.py:~PaymentWatchdog` | Stuck-payment detection threshold is hardcoded | `PaymentWatchdog` uses hardcoded time thresholds for stuck-state detection and PagerDuty alerting. The thresholds don't reference the canonical maturity constants from `constants.py` (CLASS_A=3d, CLASS_B=7d, CLASS_C=21d). A payment in `DISBURSED` state waiting for settlement should have its stuck threshold tied to the expected settlement window for its rejection class, not a generic timeout. | Import maturity constants from `constants.py` and use rejection-class-aware thresholds. A CLASS_A payment stuck for 4 days is concerning; a CLASS_C payment at day 4 is normal. |
| B5-10 | Medium | `lip/c3/state_machine_bridge.py:~warnings.warn` | `warnings.warn` for production-critical import failure | Line 38-52: Rust import failure emits `warnings.warn(...)` instead of `logging.warning(...)`. More critically, `warnings.warn` does not integrate with structured logging, log aggregation (Datadog, ELK), or alerting pipelines. A production deployment running on the Python fallback would be invisible to ops monitoring. | Use `logging.getLogger(__name__).warning(...)` with structured fields (`{"rust_available": false, "fallback": "python"}`). | B5-11 | Medium | `lip/c3_repayment_engine/rejection_taxonomy.py:is_dispute_block` | Unknown codes return False (fail-open) | `is_dispute_block(code)` returns `False` for unknown rejection codes. ## Notable positives (preserve)

- **Rust state machine is excellent.** `state_machine.rs` has an exhaustive transition table, never panics, uses `thiserror` for typed errors, and the PyO3 bindings are clean. The `is_terminal()` method and `allowed_transitions()` are exactly the right API shape. This is the gold standard for the codebase.
- **Rejection taxonomy is comprehensive.** 50+ ISO 20022 codes mapped to 4 classes with correct maturity constants matching CLAUDE.md. All 8 EPG-19 compliance-hold codes are in the BLOCK class in both Python and Rust.
- **Redis SETNX idempotency** is the correct pattern for distributed repayment dedup — the *implementation* has a fallback problem (B5-03), but the *design* is right.
- **`nav_emitter.py`** is properly thread-safe with locks and has clean per-tenant isolation.
- **Phase-aware fee decomposition** correctly distinguishes LICENSOR (30%), HYBRID (55%), and FULL_MLO (80%) take rates — the Phase 1→2→3 progression is the right commercial model.
- **`PaymentWatchdog`** with PagerDuty integration is the right operational pattern — stuck payments in production need human attention, not just logs.
- **Rust ISO 20022 parsing** handles both nested (`BkToCstmrDbtCdtNtfctn.Ntfctn[0].Ntry[0]`) and flat dict structures, with comprehensive test coverage.
- **`rejection_taxonomy.py`** constants for maturity days (`CLASS_A_MATURITY_DAYS = 3`, etc.) match CLAUDE.md exactly — no drift on the canonical values.
