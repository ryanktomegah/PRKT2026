# Batch 11 — DGEN data generation

**Scope**: `lip/dgen/` (14 files, ~5.4k LOC) — synthetic corpora for C1, C2, C3, C4, C6 + production parquet generators + validators + source-URL inspector.
**HEAD**: `2b32314`
**Reviewer focus**: Field semantics (per prior memory), reproducibility/seed handling, label leakage, EPG-19 BLOCK-code fidelity, CIPHER rule on AML typology patterns in VCS, CLAUDE.md's explicit CLASS_B labelling warning.

## Summary

DGEN is unusually thorough for a synthetic-data layer — explicit calibration citations (BIS CPMI, SWIFT GPI, ECB T2), corpus tagging for SR 11-7 lineage, seed-controlled RNGs via `np.random.default_rng`, Altman-Z derived from correlated financials (not drawn independently), and a statistical validator with chi-square priors. The **two critical issues** are both domain-semantics failures that CLAUDE.md explicitly warns against:

1. **`iso20022_payments.py:69` labels Class B as "compliance/AML hold"** — CLAUDE.md's "CLASS_B label warning" section says this is **wrong**: "CLASS_B covers systemic/processing delays only. Compliance-hold payments are BLOCK class and are never bridged." The comment was fixed in `constants.py` but the correction never propagated to DGEN. Every model trained on this corpus learns the wrong semantics of Class B.

2. **`c1_generator.py:74-82` tags `FRAU`, `LEGL`, `RR01`, `RR02` as Class B/C (bridgeable)** — these are EPG-19 compliance-hold codes that must be BLOCK class and never bridged. C1's own `synthetic_data.py` correctly filters `_BLOCK_CODES`, but DGEN's C1 generator does not — so any training run using `lip.dgen.c1_generator` (which `generate_all.py` and the production pipeline use as the canonical source) produces a corpus where EPG-19 codes carry `is_bridgeable=True`. This is a direct EPG-19 defense-in-depth violation that silently trains the classifier to bridge compliance holds.

Secondary issues: `c2_generator.py:148` calls `np.random.normal(0, 0.3)` on the **global** RNG inside `_derive_merton_dd` rather than the passed `rng` — any training run is non-reproducible on the Merton DD feature regardless of seed. The C6 generator and `aml_production.py` encode AML typology thresholds (structuring at 85-99.9% of $10k CTR, velocity at 20-80 tx/24h, layering round-trip at 93-99% return ratio) directly in source, which arguably violates the CIPHER rule against committing AML typology patterns — the rule targets the corpus file, but the generator source encodes equivalent information with similar adversarial value.

## Notable positives (preserve)

- **Calibration sources cited with URLs** — `web_inspector.py` actually HTTP GETs the BIS/ECB/SWIFT GPI pages and records accessibility. This is a real SR 11-7 traceability pattern, not documentation theatre.
- **C2 correlated financial ratios via multivariate normal** — the v1→v2 fix for "phantom correlations" is exactly the right QUANT call. The correlation matrix is domain-calibrated with documented relationships (current_ratio ↔ cash_ratio +0.72, debt_to_equity ↔ interest_coverage -0.65, etc.).
- **Altman-Z derived from features, not drawn independently** — prevents label-leakage masquerading as a feature.
- **`c4_generator.py` refuses LLM-generated training data** — explicitly calls out the circular-bias failure mode and uses templates + synonym substitution instead. The principle is right even though B11-08 notes the scale problem.
- **Corpus tagging is consistent** — every generator emits `corpus_tag`, `generation_seed`, `generation_timestamp` so every record can be traced to its source.
- **`c3_generator.py` imports the real `compute_loan_fee`** — labels are computed by the canonical fee function, not a copy (this is why B11-15 is low, not high — the coupling is intentional and correct).
- **Statistical validator has a chi-square test against priors** — not just "does it run", but "does it match the documented distribution".
- **`BICPool` hub-and-spoke topology with weighted sampling** is the right architecture for generating realistic cross-border flows (even though B11-11 flags that it's not used everywhere).
- **`c3_generator.py` uses hash-based UETR synthesis** (SHA-256 of RNG output) rather than real `uuid.uuid4()` — deterministic across seeds, CIPHER-compatible.
- **All generators use `np.random.default_rng(seed)`** — the correct modern API. The one exception (B11-03) is a bug, not a pattern.
