# Batch 7 — C6 AML Velocity (Py+Rust)

**Scope**: `lip/c6_aml_velocity/` (10 Py files, ~3.0k LOC), `lip/c6_aml_velocity/rust_velocity/` (4 Rust files, ~1.1k LOC).
**HEAD**: `2b32314`
**Reviewer focus**: AML cap enforcement correctness (EPG-16), TOCTOU races in concurrent velocity checks, salt rotation (365d + 30d overlap per CLAUDE.md), sanctions screening completeness, Decimal vs float in money arithmetic, privacy (no raw entity IDs in logs/storage).

## Summary

This is the **strongest module in the codebase**. The atomic Lua check-and-record script (EPG-25) correctly eliminates the TOCTOU race that plagues the C8 query metering (B3-01). The Rust velocity implementation mirrors the Python logic faithfully with DashMap shard-locking for thread safety. The salt rotation manager correctly implements the 365-day interval with 30-day dual-salt overlap matching CLAUDE.md. The sanctions screener uses Aho-Corasick pre-filtering + Jaccard token overlap with an honestly documented transliteration gap. The `AMLChecker` requires explicit `entity_name_resolver` configuration (EPG-24) — refusing to construct without it is exactly the right fail-closed pattern. Privacy is well-maintained: entity IDs are SHA-256 hashed before any storage or logging.

The main issues are: (1) the Rust velocity path uses `f64` arithmetic (via `amount_cents = (amount_usd * 100.0).round() as u64`) while the Python path uses `Decimal` — for amounts near the dollar cap boundary, rounding differences can cause the Rust and Python paths to make different pass/block decisions; (2) the Rust velocity checker has no Redis support, so it's in-memory only — a Rust-backed multi-replica deployment has the same per-pod isolation issue as the Python in-memory path; (3) the sanctions screener doesn't handle transliteration or phonetic matching, which is honestly documented but represents a real compliance gap for international sanctions lists.

## Notable positives (preserve)

- **Atomic Lua check-and-record (EPG-25) is the gold standard.** `_ATOMIC_CHECK_RECORD_LUA` in `velocity.py:75-137` prunes, reads, evaluates all three caps, and writes in a single `EVAL` — no TOCTOU gap in multi-worker deployments. This is the fix that B3-01 (query metering) and B2-03 (API DP budget) need but don't have.
- **VelocityChecker.check_and_record() has correct dual-path threading.** Redis path uses Lua atomics; in-memory path uses `_check_record_lock`. Both paths prevent the TOCTOU race that concurrent callers would otherwise exploit.
- **Privacy is maintained throughout.** Entity IDs are SHA-256 hashed before any storage, logging, or Redis key construction. Raw entity IDs never appear in logs (only `entity_hash[:8]`). This satisfies GDPR Art.25 data minimisation.
- **EPG-24 (entity_name_resolver) is correctly enforced.** `AMLChecker` refuses to construct without an explicit `entity_name_resolver` argument. This prevents the silent failure mode where BIC codes are passed to the sanctions screener (Jaccard score of 0.0 against all entries → sanctions gate silently disabled).
- **Salt rotation manager correctly implements CLAUDE.md constants.** `ROTATION_INTERVAL_DAYS = 365`, `OVERLAP_DAYS = 30`, Redis-backed with appropriate TTLs. Thread-safe with `_lock` on rotate.
- **Rust Aho-Corasick pre-filtering** in `sanctions.rs` is the right optimisation for large sanctions lists — O(n + m) substring scan eliminates most candidates before the O(entries) Jaccard pass.
- **Rust `check_and_record` holds the DashMap entry guard for the full check+write** — correct atomic pattern, no TOCTOU.
- **Sanctions transliteration gap is honestly documented** in code comments, audit doc references, and a test (`test_sdn_transliteration_gap`) that explicitly asserts the gap exists. This is the right way to handle a known limitation.
- **Cross-licensee aggregator correctly handles the salt overlap window** — reads aggregate across both current and previous salt keys, writes update both simultaneously.
