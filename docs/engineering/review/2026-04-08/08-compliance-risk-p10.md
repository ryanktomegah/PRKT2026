# Batch 8 — Compliance, Risk, P10 Regulatory

**Scope**: `lip/compliance/` (1 Py file, ~485 LOC), `lip/risk/` (4 Py files, ~966 LOC), `lip/p10_regulatory_data/` (16 Py files, ~3.2k LOC).
**HEAD**: `2b32314`
**Reviewer focus**: DORA/SR 11-7 artifact correctness, data-card generation, audit trail integrity, DP accounting soundness, privacy-budget composition, HHI scale consistency, re-identification attack realism, BLOCK code list drift (cross-reference with B5-01/B6-01).

## Summary

The P10 regulatory data product is the most complete non-ML component in the codebase — a full shadow pipeline from synthetic event generation through three-layer anonymization (hash → k-anonymity → differential privacy), systemic risk analytics, contagion simulation, versioned report generation with integrity hashing, and a formal privacy audit kit. The architecture is sound: constants are sourced from `lip/common/constants.py` via a local `constants.py` re-export, the pipeline is wired end-to-end in `shadow_runner.py` with per-stage timing, and the report integrity chain (SHA-256 content hash → `VersionedReport`) is correctly immutable.

The **critical issue** is a DP composition accounting error: `anonymize_batch` applies Laplace noise to three statistics per corridor (failure rate, total count, failed count) but only deducts one epsilon from the corridor budget. Sequential composition says the real privacy loss is 3ε per batch, not 1ε. The methodology documentation, privacy audit, and budget tracker all assume 1ε — so the system believes it has 3× more budget than it actually does. The `lip/risk/` module is clean — correct parametric VaR, Monte Carlo with Gaussian copula, proper Decimal arithmetic — but the `portfolio_risk.py` VaR computation bounces between Decimal and float via `math.sqrt`, losing the precision guarantee that Decimal was supposed to provide. The compliance module (`model_card_generator.py`) is a well-structured template system with good regulatory references, no issues beyond pre-3.9 typing.

## Notable positives (preserve)

- **Constants sourced from `lip/common/constants.py` via local re-export.** `p10_regulatory_data/constants.py` imports all P10 constants from the common module — single source of truth for privacy thresholds. This is the pattern the rest of the codebase should follow.
- **`VersionedReport` is frozen (`frozen=True`) with SHA-256 content hash.** Once created, the report is immutable and tamper-evident. `verify_report_integrity` correctly recomputes the hash and raises `ReportIntegrityError` on mismatch.
- **`ShadowPipelineRunner` wires the full pipeline with per-stage timing.** Collect → Flush → Anonymize → Ingest → Report → Verify, each timed in milliseconds. This is exactly the right shape for a shadow-mode integration test.
- **Contagion simulation is privacy-preserving by construction.** Corridor dependency graph uses Jaccard similarity of *bank hash sets*, not raw BICs. No raw entity identifiers enter the contagion graph.
- **`CircularExposure` is soft-flag only (CIPHER rule).** Detected circular dependency chains are `flagged_for_review=True` with no automated enforcement. Comments explicitly cite the CIPHER rule.
- **Methodology paper is auto-generated from code constants.** `generate_methodology_paper()` reads every P10 constant from `lip/common/constants.py` and embeds them in a structured document. This eliminates documentation drift for numerical parameters.
- **Monte Carlo VaR uses proper Gaussian copula (single-factor model).** `Z_i = sqrt(rho) * M + sqrt(1-rho) * epsilon_i` — textbook Li (2000) implementation. Correlated defaults via systematic factor, independent idiosyncratic risk.
- **`PrivacyBudgetTracker.deduct` raises ValueError on exhaustion.** Fail-closed budget enforcement — callers can't accidentally overspend. The `has_budget` check before deduction prevents the exception in normal flow.
- **`TelemetryCollector.flush` clears accumulators after building batches.** No double-counting across flush cycles. The HMAC signature on each batch uses deterministic JSON with `sort_keys=True`.
- **`DailyVaRReport` includes both baseline and stressed VaR.** The report structure matches what a bank risk committee expects: unstressed VaR, multiple scenarios, top concentrations, and computation time.
