# Batch 4 — C7 Execution Agent (`lip/c7_execution_agent/`)

**Scope**: 10 Python files (~2.5k LOC), 8 Go files (~1.8k LOC), 8 Rust files (~0.7k LOC) = ~5.0k LOC. Execution orchestrator, kill switch (Python + Rust + Go reader), decision log, human override, offer delivery, Go offer router.
**HEAD**: `2b32314`
**Reviewer focus**: fee floor enforcement, kill-switch soundness across 3 languages, FFI panics, offer lifecycle, thread safety, Go goroutine leaks, Rust memory ordering.

## Summary

The C7 execution agent is the most architecturally mature module reviewed so far: defense-in-depth fee floor enforcement (tiered floor recomputed at offer time), correct compliance-hold blocking (sourced from the single-source-of-truth `RejectionClass.BLOCK`), BIC-based governing law derivation (EPG-14), and a Rust+Go+Python kill switch stack with correct memory ordering and fail-closed defaults at every layer. The kill switch across all three languages is genuinely well-designed — `SeqCst` writes, `Acquire` reads, POSIX shm for cross-process visibility, and every failure mode defaults to "killed" (halt new offers).

Two findings at Critical: (1) **`aml_passed` defaults to `True`** in `process_payment` — a payment context dict missing the AML result field silently bypasses the AML gate, which is a fail-open default on a security control; (2) **UETR defaults to a random UUID** when missing — this defeats the entire GAP-04 deduplication mechanism, because two submissions of the same payment without a UETR each get a different random UETR and are treated as distinct. Both are the same class of bug: `.get(key, permissive_default)` on a field that should be required.

The human override subsystem has no thread safety, and the override sweeper accesses the override's internal `_pending` dict directly from a background thread — a concurrent modification during iteration will raise `RuntimeError` or, worse, skip entries silently. The Go offer router has a well-designed goroutine-per-offer model but the concurrency gate has a classic TOCTOU. The Rust shm layer has a non-atomic read-modify-write on the activation counter. All are fixable without architectural changes.

## Notable positives (preserve)

- **Kill switch stack is genuinely excellent.** Three languages (Rust, Go, Python), all fail-closed at every layer: Rust `AtomicBool` with correct `SeqCst`/`Acquire` ordering, POSIX shm for cross-process visibility, Go file-based reader with fail-closed on I/O error, Python bridge with fail-closed on import error. This is the right architecture for a financial safety gate.
- **`agent.py` compliance-hold gate** (lines 94-96, 304-305) sources `_COMPLIANCE_HOLD_CODES` from `get_all_codes_for_class(RejectionClass.BLOCK)` — single source of truth. No hand-maintained list. Adding a new BLOCK code to `rejection_taxonomy.py` automatically propagates to C7. This is defense-in-depth done right.
- **Tiered fee floor enforcement** at lines 510-513 recomputes `compute_tiered_fee_floor(loan_amount)` and takes `max(fee_bps, tiered_floor)` — even if C2 returns a lower fee, C7 enforces the floor. Preserves the 300 bps canonical invariant.
- **EPG-14 BIC-based governing law** (lines 564-570) correctly uses `bic_to_jurisdiction(sending_bic)` with fallback to currency only when BIC yields UNKNOWN. Matches CLAUDE.md rule.
- **`DecisionLogger`** uses `dataclasses.asdict(entry)` for canonical JSON instead of manual field enumeration — avoids the B3-02 (license token) schema-drift trap. Chain-hashing (`prev_signature`) provides DORA Art.30 tamper evidence.
- **Go `runOfferGoroutine`** select statement is clean: exactly one terminal branch, deferred cleanup, no shared-lock contention on the hot path. Channel-based state mediation is the right Go pattern.
- **`OfferDeliveryService._check_resolvable`** does a live expiry check (line 413-430) even for offers still in `_pending` — prevents accepting an offer whose expiry passed between the last sweep and the accept call. Correct boundary enforcement.
- **Rust signal handlers** use `signal_hook::flag::register` which only writes to an `AtomicBool` inside the signal handler — unconditionally async-signal-safe. No malloc, mutex, or I/O in signal context. Textbook correct.
- **`pyo3_bindings.rs`** releases the GIL via `py.allow_threads()` during atomic operations — prevents asyncio event loop + KMS monitor from blocking each other.
