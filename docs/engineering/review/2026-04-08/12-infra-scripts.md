# Batch 12 — Infrastructure & Scripts

**Scope**: `lip/infrastructure/` (Docker, K8s, Helm, monitoring), `lip/scripts/`, `requirements*.txt`, `docker-compose.yml`.
**HEAD**: `2b32314`
**Reviewer focus**: Secrets handling, container hardening, CVEs in pins, network isolation, monitoring correctness, the 94 ms SLO surfaced in code.

## Summary

Infrastructure is the cleanest batch after C6. Kubernetes manifests enforce the Pod Security Standards `restricted` profile at the namespace level, network policies are **default-deny with explicit per-component allow lists** (C4 has zero egress, C7 egress limited to Redis+DNS), containers run non-root with `readOnlyRootFilesystem` and `automountServiceAccountToken: false`, and there are **no hardcoded secrets anywhere** — everything flows through External Secrets Operator with AWS/GCP/Vault providers selectable via a single Helm value. The monitoring layer (`metrics.py`/`alerts.py`) has a dual-mode Prometheus collector (real client or in-memory fallback for tests) and a PagerDuty Events v2 integration using stdlib `urllib.request` to avoid pulling `requests` as a prod dependency. The only material issues are: (1) `requirements.txt` uses `>=` without upper bounds on most packages, which is a supply-chain risk — a compromised `cryptography` release auto-upgrades on the next `pip install`; (2) a K8s/Helm port inconsistency for Redis (network policy allows port 6380 TLS, but `docker-compose.yml` exposes 6379 plaintext — the dev/prod split is correct but undocumented and will bite anyone running the compose stack against the K8s network policy); (3) the `<AWS_ACCOUNT_ID>` placeholder in `secrets.yaml` has no CI check to reject it; (4) `Dockerfile.c1` uses `nvcr.io/nvidia/tritonserver:24.01-py3` which is a large attack surface and pinned to a 2024-01 release — Triton has had CVEs in 2024-Q2/Q3 that should be checked against this pin.

## Notable positives (preserve)

- **Pod Security Standards `restricted` profile enforced at the namespace level** — the strongest K8s hardening posture available out of the box.
- **External Secrets Operator pattern with multi-provider support** (AWS / GCP / Vault via a single Helm value). No hardcoded secrets anywhere in the repo.
- **Default-deny network policies with explicit allow lists.** C4 has zero egress (bank-side isolation), C7 egress is limited to Redis+DNS. This is rare to see done correctly.
- **All containers run non-root with `readOnlyRootFilesystem` and `automountServiceAccountToken: false`** — the full K8s hardening trifecta.
- **Dual-mode Prometheus collector** — real `prometheus_client` when available, in-memory fallback for tests. Zero test infra coupling.
- **PagerDuty alerter uses stdlib `urllib.request`** — avoids pulling `requests` as a prod dependency. Small, auditable.
- **Docker-compose decision log topic has 7-year retention configured explicitly** (SR 11-7). The intent is visible even in the local dev stack.
- **Redpanda replaces Kafka for local dev** — faster startup, same protocol. Good DX choice.
- **Helm `values-prod.yaml` overlay is minimal** — only the deltas from `values.yaml` (registry, external Redis/Kafka, GCP secrets). Clean separation.
- **`simulate_pipeline.py` has integrity checks for fee floor, royalty split, kill switch, UETR presence** — exactly the right smoke-test surface even if the C1/C2 mocks limit its scope (B12-07).
