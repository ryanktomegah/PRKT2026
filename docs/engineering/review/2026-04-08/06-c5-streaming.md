# Batch 6 ‚ÄĒ C5 Streaming (Py+Go)

**Scope**: `lip/c5_streaming/` (8 Py files, ~1.7k LOC), `lip/c5_streaming/go_consumer/` (8 Go files + 2 test files, ~1.3k LOC), `lip/c5_streaming/schemas/` (2 Avro schemas), `lip/c5_streaming/load_test/` (1 file, 265 LOC).
**HEAD**: `2b32314`
**Reviewer focus**: Kafka exactly-once semantics, at-least-once vs exactly-once confusion, backpressure, 94 ms SLO, dual-language normalization parity, EPG-19 compliance-hold propagation across languages.

## Summary

The C5 streaming layer is architecturally well-designed ‚ÄĒ two consumer implementations (Python for Flink/low-volume, Go for high-throughput) sharing the same topic topology, with good DLQ handling, proper `enable.idempotence=true` / `isolation.level=read_committed` wiring, and a clean worker-pool model in Go. The **critical systemic issue** is the same one flagged in B5-01: compliance-critical data (BLOCK codes, proprietary‚ÜíISO 20022 mappings) is now maintained in **five independent locations** across three languages (Python `event_normalizer.py`, Python `rejection_taxonomy.py`, Rust `taxonomy.rs`, Go `normalizer.go`, Go C7 `server.go`). Adding a single new compliance-hold code requires touching all five ‚ÄĒ and missing any one creates a gap in EPG-19 enforcement. The Go consumer has a **exactly-once semantics gap**: `commitOffset` uses `StoreMessage` (async auto-commit) after every message including error paths, which means failed messages are committed and never reprocessed ‚ÄĒ contradicting the docstring's claim. The Python consumer has a different but related issue: it commits on error via DLQ routing but the code comment says "not committed so it will be reprocessed" ‚ÄĒ the actual behavior depends on whether `enable.auto.commit=False` is correctly preventing auto-commit during the poll loop, which it is, but the error path still doesn't re-seek.

## Notable positives (preserve)

- **Kafka exactly-once wiring is correct at the config level.** Both Python (`enable.auto.commit=False`, `isolation.level=read_committed`, `enable.idempotence=True`, `acks=all`) and Go (same settings) have the right Kafka primitives for exactly-once. The gap is in the application-level offset commit logic (B6-02, B6-07), not the Kafka config.
- **Go worker pool model is clean.** `msgCh` with buffered channel, N workers pulling from a shared channel, context-based cancellation ‚ÄĒ textbook Go concurrency. Backpressure is natural (channel blocks when workers are busy).
- **DLQ handling is thorough.** Both Python and Go have dead-letter routing with source-topic headers, retry with exponential backoff, and produce-error tracking. The DLQ pipeline is more complete than most production Kafka consumers.
- **`CancellationDetector` is well-designed.** Thread-safe, properly locked, conservative (HUMAN_REVIEW only, no auto-blocking), with bounded memory (deque maxlen). The camt.056 attack vector documentation is honest and the detection approach is sound.
- **`StressRegimeDetector` uses injectable `time_func`.** This makes it fully testable without wall-clock dependencies. Good pattern.
- **Go normalizer tests mirror Python tests.** `normalizer_test.go` explicitly tests EPG-19 compliance-hold code passthrough, FedNow‚ÜíISO 20022 mapping, and routing-number‚Üísynthetic-BIC conversion ‚ÄĒ exactly the right regression tests for the dual-implementation.
- **`KafkaConfig.to_consumer_config` / `to_producer_config`** are clean, well-documented, and match the Go equivalents. The topic definitions with 7-year retention for decision log (SR 11-7) are the right compliance shape.
- **Proprietary rejection code normalization** (`_normalize_rejection_code`) logs unknown codes as warnings and passes them through unchanged ‚ÄĒ fail-safe rather than swallowing or mapping to a default.
